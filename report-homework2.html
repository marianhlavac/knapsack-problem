Zadání úlohy¶

  • Naprogramujte een problmu batohu:
    • metodou vtv a hranic (B&B) tak, aby omezujcm faktorem byla hodnota optimalizanho kritria. Tj. pouijte oezvn shora (pekroen kapacity batohu) i zdola (stvajc een neme bt lep ne nejlep dosud nalezen),
    • metodou dynamickho programovn (dekompozice podle kapacity nebo podle cen),
    • FPTAS algoritmem, tj. s pouitm modifikovanho dynamickho programovn s dekompozic podle ceny (pi pouit dekompozice podle kapacity nen algoritmus FPTAS).

Možné varianty řešení¶

Problm batohu je mon eit hrubou silou, heuristicky, dynamickm programovnm, algoritmem "meet-in-the-middle" a dalmi zpsoby.

V minul loze jsme se zabvali eenm hrubou silou a urychlen vpotu pomoc heuristik. V tto loze bylo vyuito dvou zpsob urychlen vpotu. Jednm ze zpsob je lep proezvn stavovho prostoru. Druhm zpsobem je vyuit dynamickho programovn a pomoc aproximanho schmatu pln polynomilnho asu (FPTAS - fully polynomial-time approx. scheme).

Popis postupu řešení¶

Algoritmus a cel program poskytujc vsledky je napsn v jazyce Rust. Program nate instance z datovch soubor urench pro tuto lohu a vypote een vemi implementovanmi metodami. Zape dlku provdn vpotu a vechna data poskytne v CSV formtu. Druhm nstrojem je Jupyter Notebook, kter poskytnut data vizualizuje.

Oproti minul lohy byl algoritmus hrubou silou pepsn z implementace pracujc s bitovmi maskami do rekurzivn varianty. Umonilo to tak jednodu implementaci Branch & Bound proezvn.

Proezvn stavovho prostoru obohacen o Branch & Bound zajist, e je prostor proezn shora i zdola. Vtve rekurze, kter by peshly kapacitu batohu, nebo by neposkytly lep vsledek, ne dosavadn dosaen jsou oezny.

Dynamick programovn umouje zrychlen vpotu na kor pamtov nronosti. Byla zvolena varianta dekompozice podle ceny, dky emu pak lze snadno implementovat FPTAS.

Konen, FPTAS pedstavuje zpsob, jak ovlivovat "kvalitu" a rychlost vpotu. Vstupn promnnou je mon definovat maximln relativn chybu. m vt je povolen tato relativn chyba, tm by ml bt vpoet teoreticky rychlej.

Kostra algoritmu¶

Cel algoritmus je k nahldnut ve zdrojovch souborech programu na GitHubu. Pro rychlou pedstavu je ne uveden ryvek z funkce majc na starost FPTAS vpoet.

...
    // Find the largest price in knapsack
    let max_price = knap.items.iter().fold(0, |acc, &x| if x.price > acc { x.price } else { acc });
    let ratio = (1.0 - accuracy) * max_price as f32 / knap.items.len() as f32;

    // Modify prices in knapsack
    let mut mod_knap = knap.clone();
    mod_knap.items = knap.items.iter().map(|item| { KnapItem { 
        id: item.id, weight: item.weight, price: f32::floor(item.price as f32 / ratio) as u16
    } }).collect();

    // Solve with dynamic solver
    let mut solution = solver_dynamic::solve(mod_knap);
...

Branch & Bound dr v pamti nejlep dosaen vsledek a ped kadm sestoupenm do vtve rekurze zkontroluje cenu zbvajcch pedmt, kter nebyly do batohu pidny. Pokud je tato cena seten s cenou batohu ni, ne nejlep dosaen vsledek, je jasn, e prohledvnm tto vtve se k lepmu vsledku ji nememe dostat a meme rekurzi na tomto mst ukonit.

Dynamick programovn pen vpoetn nronosti na pamovou nronost. Drme v pamti tabulku, do kter ukldme proveden vpoty. Konkrtn v tto implementaci jde o dekompozici podle ceny. Do tabulky tak ukldme vsledn vhy, sloupce pedstavuj ceny a dky jednotliv pedmty. Vsledek pak nalezneme na poslednm dku v nejpravjm nenulovm sloupci.

FPTAS pak pomoc formule upravujc ceny pedmt umon vloit do vpotu dal promnnou, kterou lze vpoet ovlivnit. Efektivn jde o zanedbvn uritho potu bit hodnot ceny. Vpoet je a na pozmnn ceny toton s vpotem pomoc dynamickho programovn s kompozic podle ceny a je tak pouita stejn kd implementace.

Sloupce¶

Nzvy sloupc se vyskytuj i dle v textu, zde je jejich strun popis:

  • knap_id - identifiktor instance
  • item_count - poet pedmt (konfigurace instance)
  • method - metoda vpotu
    • Recursive - rekurzivn
    • BranchAndBound - proezvn Branch & Bound
    • Dynamic - dynamickm programovnm, dekompozice podle ceny
    • FPTAS25 - FPTAS s 25% pesnost (max. rel. chybou 0.75)
    • FPTAS50 - FPTAS s 50% pesnost (max. rel. chybou 0.50)
    • FPTAS75 - FPTAS s 75% pesnost (max. rel. chybou 0.25)
  • price - vypoten celkov cena batohu
  • elapsed_ms - doba vpotu v milisekundch
  • optimal_price - optimln cena batohu

Porovnání relativní chyby u FPTAS¶

Na grafu ne lze vidt jak byly relativn chyby pro jednotliv pesnosti (max. relativn chyby) algoritmu FPTAS.

Sloupeek expected pedstavuje oekvanou maximln relativn chybu. Maximum a minimum jsou krajn ppady a mean je prmrn relativn chyba.

In [174]:
pd.options.mode.chained_assignment = None

# Vpoet absolutnch a relativnch chyb u kadho vpotu
data_werr = data
data_werr['error'] = data_werr['optimal_price'] - data_werr['price']
data_werr['relative_error_%'] = (data_werr['error'] / data_werr['optimal_price']) * 100

expected = { 'FPTAS25': 75, 'FPTAS50': 50, 'FPTAS75': 25 }
minimum = { 'FPTAS25': 1, 'FPTAS50': 1, 'FPTAS75': 1 }
maximum = { 'FPTAS25': 0, 'FPTAS50': 0, 'FPTAS75': 0 }
means = { 'FPTAS25': 0, 'FPTAS50': 0, 'FPTAS75': 0 }
samples = { 'FPTAS25': 0, 'FPTAS50': 0, 'FPTAS75': 0 }

for index, row in data_werr.iterrows():
    if row['method'].startswith('FPTAS'):
        means[row['method']] += row['relative_error_%']
        samples[row['method']] += 1
        
        if row['relative_error_%'] < minimum[row['method']]:
            minimum[row['method']] = row['relative_error_%']
        
        if row['relative_error_%'] > maximum[row['method']]:
            maximum[row['method']] = row['relative_error_%']
            
for index, row in means.items():
    means[index] = row / samples[index]

relative_errors = pd.DataFrame({
    'minimum_rel_err_%': minimum,
    'mean_rel_err_%': means,
    'expected_rel_err_%': expected,
    'maximum_rel_err_%': maximum
})

Závěr¶

Urychlen vpotu pomoc proezvn a dynamickm programovnm bylo spn. Zrychlen vpotu je daleko lep, ne zrychlen, kterho se podailo doshnout v minul loze.

U FPTAS se ovem u takovho spchu pravdpodobn nedoshlo. Zrychlen oproti ostatnm metodm nen znateln, naopak se v nkterch ppadech ukzala asov nronost jako vy.

Nkter vpoty byly vypoteny skuten s chybami, tyto chyby se prmrn pohybovaly kolem 2,6% (relativn chyba). Dalo by se vak oekvat, e chyby budou daleko vt a vce zce koprovat stanovenou hranici maximln relativn chyby. Zrove se oekvalo dal vrazn zrychlen.

Vzhledem k tomu, e jsme byli schopn nalzt optimum v podobnch asech, se FPTAS v tomto ppad nevyplc.
